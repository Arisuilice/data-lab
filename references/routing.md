# routing

本文件用于判断数据分析任务该走多轻或多重的流程。

## 1. 默认路线

默认选择 Standard：一个主脚本、一份报告、必要表图。

```text
load data → profile → clean minimally → baseline → improve → interpret → report
```

## 2. 任务分级

### Quick

适用：

- 用户只问一个局部问题；
- 不需要正式报告；
- 数据较小，结果可以直接在对话中展示。

产物：可无文件，或只生成一张图/一张表。

### Standard

适用：

- 一个数据集；
- 一个目标变量或一个研究方向；
- 需要报告或可复现脚本。

产物：

```text
scripts/analyze.py
outputs/report.md
outputs/run_summary.json
outputs/tables/*
outputs/figures/*
```

### Expanded

适用：

- 多个目标变量；
- 多个相互独立问题；
- 多数据源但仍是一次性交付；
- 单脚本会变得难以阅读。

产物：2-5 个脚本，按问题拆，不建 DAG。

### Project

适用：

- 多轮长期项目；
- 生产或合规审计；
- 需要多人协作、回滚、版本化；
- 用户明确要求严格流水线。

才允许创建状态、checkpoint、详细审计和更复杂目录。

## 3. 拆分原则

拆分必须服务于：

- 更高正确性；
- 更快定位错误；
- 更清晰复现；
- 更稳定运行。

不要为了“符合流程”拆分。

## 4. 不追问原则

优先推进。只有以下情况需要问用户：

- 目标变量完全无法判断；
- 同一数据可支持多个互斥目标，且选择会改变全部分析；
- 涉及高风险删除、脱敏或外推，用户意图不明确；
- 用户要求的交付格式无法推断。

其他情况先做合理假设，并在报告中说明。

## 5. 升级/降级

运行中可升级或降级：

- 如果模型效果差，不升级流程，先升级模型与诊断；
- 如果脚本太长，拆脚本；
- 如果任务目标发散，先收敛到用户最关心的问题；
- 如果出现流程性产物多但发现少，立即降级到核心分析。
